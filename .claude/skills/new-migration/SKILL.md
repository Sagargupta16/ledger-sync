# Adding or changing a database model in ledger-sync

## The empty-downgrade convention (read this first)

Since 2026-02-03, **every Alembic migration in this project has an empty `downgrade()`**. Recovery is via DB backup, not down-migration. This is intentional — many migrations contain data backfills that aren't reversible (you can't un-deduplicate, you can't un-merge), so promising rollback would be a lie.

```python
def downgrade() -> None:
    # Per project convention: rollback via DB backup, not down-migration.
    pass
```

Don't write functional `downgrade()` logic. CI doesn't enforce this yet (debt #5 from the audit), but reviewers will reject it.

## Steps

1. **Add or modify the model in [db/_models/<domain>.py](backend/src/ledger_sync/db/_models/).** Domains:
   - `user.py` — User, UserPreferences, AuditLog
   - `transactions.py` — Transaction, ImportLog, AccountClassification
   - `analytics.py` — pre-aggregated tables (DailySummary, MonthlySummary, etc.)
   - `investments.py` — InvestmentHolding, NetWorthSnapshot, TaxRecord
   - `planning.py` — Anomaly, Budget, FinancialGoal, RecurringTransaction, ScheduledTransaction
   - `enums.py` — only Python enums

   New table doesn't fit anywhere? Discuss before creating a new file — usually it does fit.

2. **Mandatory fields on every user-data table:**
   ```python
   user_id: Mapped[int] = mapped_column(
       ForeignKey("users.id", ondelete="CASCADE"),
       index=True,
       nullable=False,
   )
   ```
   `ondelete=CASCADE` because deleting a user wipes their data. `nullable=False` because user-scoping is non-negotiable. `index=True` because every query filters by it.

3. **Composite index on `(user_id, date)`** for time-series tables — every analytics query starts with this filter and ordering. Without it, queries scan the whole user partition.

4. **Re-export through the facade** ([db/models.py](backend/src/ledger_sync/db/models.py)):
   ```python
   from ledger_sync.db._models.<domain> import MyNewModel
   __all__ = [..., "MyNewModel"]
   ```
   Consumers import `from ledger_sync.db.models import X` — never directly from `_models/`. The facade ensures SQLAlchemy registers all tables before `Base.metadata.create_all()` and lets us reorganize `_models/` without breaking call sites.

5. **Generate the migration:**
   ```bash
   cd backend
   uv run alembic revision --autogenerate -m "<short slug>"
   ```
   Review the generated file. Autogenerate is good but not perfect — common things to fix:
   - Index renames it didn't notice
   - `op.drop_column` followed by `op.add_column` (means it didn't recognize a rename — fix manually with `op.alter_column`)
   - JSON column defaults (autogenerate often gets these wrong; spot-check)

6. **Replace the autogenerated `downgrade()` with `pass`** (see top of this file).

7. **Apply locally:**
   ```bash
   uv run alembic upgrade head
   ```
   Verify the schema with a quick `sqlite3 ledger_sync.db .schema` or a tool that connects.

8. **Run the test suite** — the integration tests will catch user-scoping regressions:
   ```bash
   uv run pytest tests/
   ```

9. **CI auto-deploys.** Migrations run against production on push to `main` via [.github/workflows/migrate.yml](.github/workflows/migrate.yml), but only when the changed paths match `backend/alembic/**` or `backend/src/ledger_sync/db/**`. So just merging the PR runs it. Have a Neon backup ready.

## Common pitfalls

- **Adding a NOT NULL column to a populated table** — needs a backfill step or a default. Autogenerate doesn't add either. Pattern:
  ```python
  # Step 1: add as nullable
  op.add_column('table', sa.Column('new_col', sa.String(), nullable=True))
  # Step 2: backfill
  op.execute("UPDATE table SET new_col = '...' WHERE new_col IS NULL")
  # Step 3: alter to NOT NULL
  op.alter_column('table', 'new_col', nullable=False)
  ```

- **Dropping a column** — irreversible without backup, so don't do this lightly. Soft-deprecate first (stop writing, migrate readers, then drop in a later PR).

- **Renaming a column** — autogenerate sees this as drop+add. Use `op.alter_column('table', 'old_name', new_column_name='new_name')` manually.

- **JSON column shape changes** — Alembic doesn't validate JSON shapes. If you add a required field to a JSON pref (e.g. `UserPreferences.growth_assumptions`), backfill explicitly or write the consumer to handle missing keys.

- **SQLite-specific raw SQL in `op.execute()`** — older migrations have these; new ones must work on Postgres too. If you must use raw SQL, use SQLAlchemy textual constructs that compile to both dialects.

## Frontend follow-up

If you added a column the frontend reads, the frontend type lags by exactly one PR — see the **schema-drift-check** skill. The drift is silent.

## Definition of done

- [ ] Model in the right `_models/<domain>.py`
- [ ] `user_id` FK with `ondelete=CASCADE` and `index=True` (if user-data)
- [ ] Re-exported through `db/models.py` facade
- [ ] Migration generated, autogenerate output reviewed
- [ ] **`downgrade()` replaced with `pass`**
- [ ] Applied locally, schema spot-checked
- [ ] `uv run pytest` passes
- [ ] Frontend types updated if response shape will change
